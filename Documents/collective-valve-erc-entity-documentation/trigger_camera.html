    trigger_camera
    Half-Life trigger entity
    description

    The trigger_camera entity allows you to remove control from the player for a brief period of time, in which period you can manipulate the camera view.

    usage

    The trigger_camera entity allows you to remove control from the player for a brief period of time, in which period you can manipulate the camera view. There are a few different ways you can use the camera.

    1. Fixed Eye, Fixed Focus
      This is about as easy a camera as you can make. You would generally use this when you have an event in a separate area that you want the player to witness.

      1. Place the trigger_camera. It will be used as the "eye" so position it wherever you'd like the view to be from.
      2. Place an info_target in the area you want the camera to be looking at. The camera will look at this exact spot. Give the info_target a Name (targetname).

          Note: you do not need to use an info_target for the camera's target. You could use any entity such as a weapon, item, or monster as well.
      3. In the trigger_camera properties, give the Target field a value equal to the name of the info_target. Give the camera a unique Name (targetname).
      4. Set the trigger_camera Hold Time (wait). The hold time determines how long the camera period will last before control is returned to the player.
      5. In the trigger_camera Flag properties, place a check next to 'Freeze Player'. This will stop the player from moving around while the camera sequence is being played out.
      When the trigger_camera's targetname is triggered, the player's view will be transfered to that of the camera's for a time equal to the Hold Time value.

    2. Fixed Eye, Moving Focus
      The fixed eye/moving focus method is useful for things like imitating security cameras, or tracking a creature's movements with a camera. The setup is similar to the above.

      1. Create a camera, as above.
      2. Create the moving focus. Generally, there are two ways to do this: you can create a monster that the camera will follow, or you can create a func_train. Either way, the focus must have a Name (targetname) for the trigger_camera to target.

          Note: If you use a func_train as the focus, the func_train must have an origin brush included in its construction. An origin brush is a brush completely covered in the ORIGIN texture, and it must be a part of the entity. As well, if you don't want the func_train to be visible, the Render Mode (rendermode) should be set to Texture (2) and the FX Amount (renderamt) should be set to 0. Last, set the Not Solid flag in the Flags properties.
      3. In the trigger_camera properties, give the camera a target value equal to the name of the focus, as set above.
      As above, the camera must be triggered to start. If the focus entity starts stationary, it should be triggered at this point as well.

    3. Moving Eye, Fixed Focus
      The moving eye/fixed focus is good for creating dramatic effect, or anything where you want to move the camera around while maintaining view of a particulary object or creature.

      1. Create a camera, as above (following directions for type 1).
      2. Build a set of path_corners that form a path in the camera area, linking each to the next like for a func_train.
      3. In the trigger_camera properties, in the Path Corner (moveto) field, enter the Name (targetname) of the first path_corner, where the trigger_camera is to move initially.

          Note: the camera view will initially start at the trigger_camera's position and move to the first path corner at the speed defined in the Initial Speed (speed) field.
      Like type 1 camera, only the camera needs to be triggered for this to take effect.

        Note: The trigger_camera's Freeze Player flag must be checked if you want the trigger_camera to follow a set of path_corners.
    4. Moving Eye, Moving Focus
      The moving eye/moving focus camera is useful for "taking" the player somewhere. It involves moving the eye and the focus at the same time.

      1. Create a camera, as above (following directions for type 3).
      2. Create the moving focus. Generally, there are two ways to do this: you can create a monster that the camera will follow, or you can create a func_train. Either way, the focus must have a Name (targetname) for the trigger_camera to target.

          Note: If you use a func_train as the focus, the func_train must have an origin brush included in its construction. An origin brush is a brush completely covered in the ORIGIN texture, and it must be a part of the entity. As well, if you don't want the func_train to be visible, the Render Mode (rendermode) should be set to Texture (2) and the FX Amount (renderfx) should be set to 0. Last, set the Not Solid flag in the Flags properties.
      3. In the trigger_camera properties, give the camera a target value equal to the name of the focus, as set above.

          Note: The trigger_camera can be connected to the same set of path_corners that the func_train follows. In this case, it is a good idea to either start the func_train one path_corner ahead of the trigger_camera, or to start the func_train moving shortly before the trigger_camera is triggered.
      When using this type of camera, you have to bear in mind the respective speeds of the func_train and trigger_camera. If the camera passes the train, the view will be flipped around, and unless this is the effect you're going for, it should probably be avoided.
    additional notes

    There are several things you should take note of while building camera sequences.

    • The camera area must be in the player's PVS (potentially visible set) otherwise, to the engine, the area will not exist. To get around this, I use the following method:

        Fade the view out using an env_fade. Use a trigger_teleport to teleport the player to the new camera area. (Make the trigger_teleport use a multisource, then trigger the multisource on, then off 0.1 seconds later using a multi_manager) Make sure the corresponding info_teleport_destination's angle is aimed at the camera area. Fade the view back in using another env_fade.
    • Going with the above, if it is inconvenient to have the player in the area of the camera (ie: if you don't want him visible, and you can't hide him from camera view) the player can be placed behind a brush entity like a func_wall or func_illusionary. It is only important that the engine be drawing the area that the camera is viewing (so make sure the player is facing the right way!)
    • Lock him down! In the trigger_camera spawnflag properties, the "Freeze Player" flag should almost always be checked, unless you're trying to simulate some type of third person view point for part of the game. Also remember that the "Freeze Player" flag must be checked if you are using a trigger_camera that follows a set of path_corners.
    • Using an env_fade to fade the view in and out when the camera starts and ends is a matter of esthetics. I prefer this method rather than just cutting directly to and from the camera view.
    • Camera positioning is not an exact science. If the camera is following a path, and the path veers too close to a wall, the camera can sometimes clip through the wall causing viewing errors including apparent screen freezes and HOM (hall of mirrors) effects. This can be corrected by repositioning the path.
    Note: The example map starts out with a camera of type 4, then fades into a camera of type 3. Note that the second camera can almost be considered a type 4, but its focus, a monster_gman, is not made to move around.

    properties

  • acceleration (string) Acceleration units/sec^2 (default: 500)
  • deceleration (string) Stop Deceleration units/sec^2 (default: 500)
  • moveto (string) Path Corner
  • speed (string) Initial Speed (default: 0)
  • wait (integer) Hold time (default: 10)
  • delay (string) Delay before trigger (default: 0)
  • killtarget (target_destination) KillTarget
  • targetname (target_source) Name
  • target (target_destination) Target

    • Flags
    This entity has the following flags. Flags can be either on or off. In the editor, these flags can be accessed in the Flags section of the entity properties.

  • 1 - Start At Player - The camera will start at the player's eye position before moving anywhere.
  • 2 - Follow Player - The camera will use the player as a focal point.
  • 4 - Freeze Player - The player will not be able to move or take any action during the camera period.
